Задание
Кейс (синтетика). Вы ведёте проект внедрения DWH и BI-отчётности для МегаБайт (~120 магазинов, 5 регионов, PostgreSQL + Power BI). Заказчик считает, что цифры в витрине «поползли».

Письмо заказчика (используйте как вход):

*Тема: Эскалация — расхождение витрины с POS, магазины M-104, M-217, M-089*

Добрый день,

С 14.04 витрина дневных продаж не сходится с кассовой системой по трём точкам. По валовой выручке за 16.04 расхождение до 4,7 % — на таком уровне мы не можем использовать отчёт для заказа фреша.

Нам нужно письменно до 18:00 18.04 (МСК): (1) подтвердите, рассматриваете ли вы инцидент как проблему на стороне DWH/витрины; (2) гипотеза причины или перечень проверок; (3) срок согласования цифр с POS.

С уважением, Ирина Соколова, Директор по операциям МегаБайт *(синтетика)*

Задача: письмо-ответ до ~450 слов: признание ситуации без ложного признания вины, что уже проверили (2–4 пункта), что запрашиваете у заказчика, ближайшие шаги с датами.

Роль аналитик: добавьте в письмо блок «Технические гипотезы расхождения» — ≥3 гипотезы (дубли при повторной загрузке, расхождение TZ между POS и DWH, пропуск возвратов) с методом проверки для каждой (конкретный SQL / действие).
Роль sales: добавьте второй файл — «внутренняя заметка аккаунту» (8–12 строк): как не потерять отношения.

Как распределить работу с LLM:
- 🤖 LLM: генерация черновика письма-ответа, структура шагов с датами.
- ✍️ Вручную: убрать «извинения за неудобства» и другие шаблонные фразы, проверить что тон деловой без ложных обещаний, добавить конкретные проверки.

В README.md обязательно:
• Выбранный путь (А/Б) и обоснование
• Полный текст ≥1 ключевого промпта
• ≥3 конкретных правки после LLM с пояснением «почему»
• Краткое описание каждого файла в архиве

В README пометьте: «Путь А: реальный (обезличен)» или «Путь Б: синтетика».
Что на выходе
Время: 30–60 мин

Артефакты: Архив ZIP ≤ 18 МБ:
• `README.md` — путь (А/Б), промпт(ы) для LLM, ≥3 правки после LLM
• `reply.md` — письмо-ответ на эскалацию (≤450 слов)
• Опционально: `internal_note.md` — внутренняя заметка аккаунту (для роли Sales)

Критерии приёмки

• Ответ оформлен как письмо (тема + приветствие + подпись).
• Пункты (1)–(3) закрыты по сути.
• >=1 измеримая промежуточная веха (дата/время).
• Нет фраз-клише («приносим извинения за неудобства», «в кратчайшие сроки»); нет обещаний без даты; нет признания вины без фактов.
• Источник данных (→ score_business): свои реальные данные из проекта (обезличены) → макс. балл; собственная синтетика → стандартный балл.
Чему вы научитесь
• Использовать role prompting («ты — PM проекта DWH, отвечаешь COO») для генерации письма в правильном тоне и контексте.
• Детектировать шаблонность LLM: модель по умолчанию пишет «приносим извинения за неудобства» — уметь вычищать клише и заменять конкретикой.
• Применять iterative refinement: черновик -> «убери воду, добавь даты» -> «сделай тон увереннее» -> финал.
• Знать границу AI в деловой коммуникации: LLM не знает реальный контекст проекта — факты, сроки и обещания всегда на человеке.
Подсказки
Письмо заказчика уже встроено в задание — дополнительный стартовый файл не нужен.

Как работать с LLM:
1. Подайте на вход: письмо Ирины Соколовой + контекст проекта МегаБайт (DWH, витрина дневных продаж, PostgreSQL) + вашу роль (руководитель проекта).
2. Попросите: «Напиши деловой ответ на эскалацию: признай ситуацию без вины, перечисли 3 проверки, запроси данные у заказчика, дай шаги с датами. Без шаблонных извинений».
3. Проверьте вручную: нет ли ложных обещаний («всё исправим к завтра»), адекватны ли сроки, корректен ли тон (не заискивающий, не агрессивный).
4. Доработайте: уберите «воду» и шаблонные фразы, добавьте конкретные технические проверки, уточните даты.

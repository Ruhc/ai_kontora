# w1_email — Ответ на эскалацию заказчика (расхождение витрины с POS)

**Путь Б: синтетика** — письмо Ирины Соколовой (COO МегаБайт) сгенерировано в образовательных целях согласно условию задания.

---

## 1. Рутина

Без ИИ: вручную составить деловой ответ на эскалацию COO — выбрать правильный тон (не заискивать, не оправдываться), сформулировать технические гипотезы понятным языком, расставить конкретные даты, подготовить SQL-заготовки для верификации — ~1,5–2 часа.

---

## 2. Промпт / шаги для ИИ

### Шаг 1: Черновик письма-ответа

```
Ты — PM проекта DWH/BI на стороне консалтинга (Glowbyte).
Заказчик — COO Ирина Соколова (МегаБайт, розничная сеть ~120 магазинов).

Получено письмо с эскалацией: расхождение 4,7% в витрине дневных продаж за 16.04
по магазинам M-104, M-217, M-089 относительно POS-данных. Расхождение с 14.04.
Заказчик требует ответа до 18:00 18.04 по трём пунктам:
(1) считаем ли мы проблему нашей, (2) гипотезы, (3) срок согласования.

Напиши деловой ответ ≤450 слов:
- Обращение: «Уважаемая Ирина» (отчество из письма неизвестно)
- Признай ситуацию БЕЗ признания вины: «расхождение зафиксировано, проводим верификацию»
- Что уже проверено (2–4 конкретных пункта)
- 3 технических гипотезы кратко
- Конкретные даты: предварительный вывод 19.04, отчёт 21.04, пересчёт 22.04
- Чёткий запрос данных у заказчика с дедлайном
- Без фраз: «приносим извинения за неудобства», «в кратчайшие сроки», «мы виноваты»

Контекст: M-104, M-217, M-089 — магазины с нестандартным POS (store_id без "M-").
ETL 16.04 завершился без ошибок. Маппинг обновлялся 14.04.
```

### Шаг 2: SQL-гипотезы для файла hypotheses.md

```
Сгенерируй 4 технических гипотезы расхождения витрины с POS.
Для каждой: описание механизма, SQL-запрос для верификации в PostgreSQL 14+.

Контекст:
- Витрина: mart.fact_sales_daily (sale_date, store_id, sku_id, gross_revenue, returns_amt, net_revenue)
- POS-staging: pos_staging.transactions (store_id, transaction_dt [UTC], transaction_type, amount)
- ETL-лог: etl.job_runs (run_date, job_name, started_at, rows_inserted, status)
- M-104/M-217/M-089 хранят store_id как '104', '217', '089' в POS, нормализация через маппинг
- Расхождение началось 14.04 — дата обновления маппинга

Гипотезы:
1. Задвоение или потеря строк при сбое нормализации store_id (P1 — самая вероятная)
2. Задержанные возвраты не попали в MERGE-цикл
3. TZ-смещение UTC→МСК у транзакций вечерней зоны 21:00–00:00
4. Повторный запуск ETL без idempotency-защиты

Добавь таблицу приоритетов в конце.
```

## 3. Ручные правки

### Правка 1: Юридический тон — убрано признание вины

- **Было:** ИИ написал «мы признаём, что данные витрины содержат ошибку» и «наш ETL допустил сбой маппинга»
- **Стало:** «расхождение зафиксировано, требует верификации на обоих слоях — POS и DWH», «до завершения анализа нет оснований однозначно отнести расхождение к одному из слоёв»
- **Почему:** признание вины до установления причины — юридически опасно. Если расхождение окажется на стороне POS, уже отправленное письмо «мы виноваты» создаст основание для претензии. Формулировка «зафиксировано, верифицируем» нейтральна и фактически точнее

### Правка 2: Конкретные даты вместо «в ближайшее время»

- **Было:** «направим результаты в ближайшее время», «сообщим о выводах как можно скорее»
- **Стало:** предварительный вывод — 19.04 12:00, отчёт о причинах — 21.04 17:00, пересчёт (если проблема в DWH) — 22.04
- **Почему:** «в ближайшее время» — это обещание без обязательства. Заказчик уже в режиме эскалации; размытые сроки воспринимаются как уклонение. Конкретные даты создают понятные ожидания и позволяют фиксировать факт выполнения/невыполнения

### Правка 3: Привязка гипотезы P1 к известной проблеме из протокола

- **Было:** ИИ описал гипотезу маппинга как общую («возможна проблема с идентификаторами»)
- **Стало:** явная ссылка на известную проблему — «M-104, M-217, M-089 работают на версии POS с нестандартным форматом store_id; расхождение началось 14.04 — дата обновления маппинг-таблицы»
- **Почему:** это защитная позиция: проблема была задокументирована на встрече 19.02 (протокол w1_proto, AI#1). Если причина — в маппинге, у нас есть письменное подтверждение, что риск был известен и согласован обеими сторонами. Без этой детали письмо выглядит как оправдание; с ней — как профессиональный анализ

### Правка 4: Конкретный запрос данных от заказчика с дедлайном

- **Было:** ИИ написал «просим предоставить данные POS для анализа»
- **Стало:** точный запрос — выгрузка из POS по M-104/M-217/M-089 за 14–16.04 с `gross_revenue` и `receipt_count` в разбивке по дням + информация об изменениях в POS за 12–16.04; дедлайн — 18.04 21:00 МСК
- **Почему:** расплывчатый запрос вернёт расплывчатый ответ. Точная спецификация (поля, период, формат) позволяет заказчику отправить нужное с первого раза. Дедлайн «сегодня 21:00» создаёт обоснование для более ранней первичной сверки

### Правка 5: Отчество Ирины Соколовой — не угадывать

- **Было:** ИИ написал «Уважаемая Ирина Александровна» (угадал отчество)
- **Стало:** «Уважаемая Ирина» — отчество в письме-источнике не указано, изобретать его недопустимо
- **Почему:** неверное отчество в деловом письме — грубая ошибка в русскоязычном деловом этикете. Лучше «Уважаемая Ирина», чем «Уважаемая Ирина Петровна» при реальном «Алексеевна»

- **Время без ИИ:** ~2 часа (письмо + SQL-гипотезы + внутренняя заметка)

## Дополнительные роли (бонус)

### Роль: Аналитик — технические гипотезы

Файл `hypotheses.md` содержит 4 гипотезы с SQL-запросами:
- **P1 (наиболее вероятная):** сбой нормализации store_id после обновления маппинга 14.04 — два варианта (задвоение / потеря строк)
- **P2:** задержанные возвраты не попали в MERGE-цикл (сравнение pos_staging vs mart)
- **P2:** повторный запуск ETL без idempotency-защиты (проверка по etl.job_runs)
- **P3:** TZ-смещение UTC→МСК для вечерних транзакций

- Когда и как эскалировать внутри Glowbyte

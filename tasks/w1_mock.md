Задание
Контекст: МегаБайт — розничная сеть, ~120 магазинов, 5 регионов. Дмитрий Волков (руководитель розницы) хочет видеть операционный дашборд дневных продаж.

Вход (один из трёх вариантов или микс):
1) Структурированные заметки интервью с Дмитрием Волковым *(синтетика / LLM)*;
2) Паспорт дашборда (цели, аудитория, KPI, частота, ограничения);
3) Синтетический датасет / описание полей + 1 абзац «какие решения принимает пользователь».

Это шаг 2 из 6-шагового процесса разработки дашборда, принятого у BI-интеграторов: бизнес-исследование → макет (Figma) → подготовка данных → рабочий прототип → тестирование → продакшн.

Результат: один макет дашборда — экспорт из Figma (PNG/PDF) или ссылка на файл Figma с доступом на чтение. Допускается также v0.dev, низкопольный HTML.

Обязательно на макете: KPI-полоса (3–5 карточек), минимум один график или таблица, фильтры (период, магазин/регион). Названия метрик должны совпадать с входом.

Роль аналитик: оформите паспорт дашборда как отдельный документ — формализованная карточка: название дашборда, владелец (бизнес-заказчик), аудитория и роли доступа, перечень KPI с определениями, источники данных (таблицы витрины), частота обновления, SLA готовности, ограничения (объём данных, производительность), дата согласования. Паспорт — стандартный артефакт аналитика в проектах DWH/BI.
Роль BI: проверьте, что метрики на макете соответствуют тому, что реально можно построить в Power BI / Fine BI / Visiology. Укажите тип визуала (card, bar chart, matrix...).
Роль маркетинг: оцените макет глазами клиента — понятно ли с первого взгляда ценностное предложение дашборда.

Как распределить работу с LLM:
- 🤖 LLM: генерация паспорта дашборда, текста KPI-карточек, структуры макета.
- ✍️ Вручную: проверить что метрики совпадают с источниками, скорректировать визуальную иерархию, убрать нереалистичные визуализации.

В README.md обязательно:
• Выбранный путь (А/Б) и обоснование
• Исходный вход (бриф/интервью)
• Полный текст ≥1 ключевого промпта
• ≥3 конкретных правки после LLM с пояснением «почему»
• Краткое описание каждого файла в архиве
• Дизайн-решения (≥2 пункта)

Входные данные — два пути:
(А) *Реальный бриф* — из рабочего проекта с обезличиванием. (Б) *Синтетика* — `starters/w1_mock_brief.md` или LLM. В README пометьте путь.

Артефакты: Архив ZIP ≤ 18 МБ:

• `dashboard_mock.png` или `.pdf` — экспорт макета дашборда
• `dashboard_passport.md` — паспорт дашборда (роль Аналитик): название, владелец, аудитория, KPI, источники, SLA, ограничения
• Опционально: ссылка на Figma-файл с доступом на чтение

Критерии приёмки

• Виден один связный экран дашборда (KPI-полоса + график/таблица + фильтры).
• В README зафиксированы вход и текст промпта.
• >=2 пункта про ограничения (контраст, плотность, мобильность).
• Нет реальных логотипов/данных клиентов.
• Источник данных (→ score_business): свои реальные данные из проекта (обезличены) → макс. балл; собственная синтетика → стандартный балл; стартовый файл без доработки → базовый балл.
Чему вы научитесь
• Использовать role prompting (<<ты UX-дизайнер BI-дашбордов>>) для получения макета с правильной визуальной иерархией.
• Критически оценивать AI-макет: LLM предлагает красивые, но нереализуемые визуализации — уметь отличать реальное от галлюцинации.
• Фиксировать в README цикл «промпт -> результат -> ручная правка -> почему», превращая разовый эксперимент в воспроизводимый воркфлоу.
• Связывать метрики макета с реальными полями витрины — LLM не знает вашу схему данных, это всегда ручная сверка.

1. Подайте на вход: описание аудитории (Дмитрий Волков, руководитель розницы), список KPI, частоту обновления.
2. Попросите: «Сгенерируй паспорт дашборда и опиши layout: KPI-полоса сверху, график тренда, таблица топ-магазинов, фильтры слева».

4. Доработайте: уберите нереалистичные элементы (3D-графики, анимации), скорректируйте иерархию, добавьте недостающие фильтры.

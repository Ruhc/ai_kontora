## Обсуждённые темы

### 1. Бизнес-цель и метрики дашборда

Операционные менеджеры получают данные по продажам с задержкой 2–3 дня через ручной процесс: аналитик выгружает из 1С → склеивает в Excel → отправляет по почте. Цель — дашборд, доступный ежедневно к 08:00 МСК с результатами дня D-1.

Согласован минимальный набор KPI:
- Выручка (факт)
- Количество чеков
- Средний чек
- Выполнение плана (факт vs план)
- Доля возвратов — инициатива Алексея Жукова: в части магазинов возвраты достигают 5%, информация поступает только в конце месяца

### 2. Источники данных и архитектура ETL

**POS-система:** основной источник транзакций. Данные попадают в PostgreSQL-реплику один раз в сутки, около 02:00 МСК.

**1С:** справочники dim_store и dim_sku — забор через CDC.

**Проблема store_id:** магазины M-089, M-312 и M-055 работают на другой версии кассового ПО. Их store_id передаётся как числовое значение без префикса «M-» (например, «089» вместо «M-089»). Требуется маппинг в ETL для нормализации к единому формату. Магазин M-055 открылся в январе 2024 — исторических данных в POS нет.

**SLA ETL:** для готовности данных к 08:00 МСК запуск ETL запланирован на 06:30 МСК. [контекст]

**Окно MERGE:** для магазинов с задержкой корректировок возвратов (включая M-089) установлено окно инкрементальной загрузки 3 дня — защита от поздних корректировок без потери уже загруженных строк. [контекст]

### 3. Загрузка планов из Excel

Категорийные менеджеры формируют месячный план в Excel. Файл размещается на шаре `\\fs01\plans\` до 5-го числа каждого месяца.

**Проблема:** формат файла нестабилен — периодически добавляются столбцы или меняется название листа. Согласовано: при загрузке выполнять валидацию по утверждённому шаблону; при несоответствии — автоматическое оповещение ответственному для ручного разбора.

### 4. Исторические данные

**Позиция Алексея Жукова:** данные 1С за прошлый год содержат ошибки; очистка — отдельный проект на 1+ месяц, не вписывается в бюджет.

**Позиция Дмитрия Кравцова:** руководство ожидает сравнение год к году (YoY). Без исторических данных этот сценарий недоступен.

**Итог:** стороны не пришли к единому решению на встрече. Дмитрий инициировал компромисс: MVP запустить без истории, вернуться к вопросу после релиза.

### 5. Разграничение доступа (RLS)

Данные по выручке — конфиденциальны. Два уровня ограничений:

1. **По роли:** дашборд доступен только сотрудникам с ролью «Операционный менеджер» и выше.
2. **По региону:** менеджер одного региона не видит данные другого. Реализуется через справочник «пользователь — регион», который предоставит Алексей Жуков до конца текущей недели.

## Принятые решения

1. **Состав MVP-витрины:** fact_sales_daily (гранулярность: день × магазин × SKU), fact_plan_monthly, dim_store, dim_sku.
2. **KPI дашборда (5 показателей):** выручка, выполнение плана, средний чек, доля возвратов, топ-5 SKU по выручке.
3. **SLA обновления:** данные готовы к 08:00 МСК ежедневно; ETL стартует в 06:30 МСК. [контекст]
4. **MVP без исторических данных 1С:** загрузка исторических данных выходит за рамки текущего бюджета и дедлайна 01.04.2024. Вопрос YoY-сравнения — в backlog после запуска.
5. **Нормализация store_id в ETL:** все store_id приводятся к формату «M-NNN» на стороне ETL; затронуты магазины M-089, M-312, M-055.
6. **Валидация Excel-плана:** при загрузке проверяется соответствие утверждённому шаблону; сбой валидации → автоматическое оповещение и ручной разбор.
7. **RLS по регионам:** реализуется на уровне BI через справочник «пользователь — регион».
8. **Окно MERGE 3 дня для fact_sales_daily:** защита от поздних корректировок возвратов из POS-систем проблемных магазинов. [контекст]

## Открытые вопросы

1. **Загрузка исторических данных из 1С:** руководство МегаБайт ожидает YoY-сравнение, но качество данных 1С требует отдельного проекта по очистке (~1 месяц). Нужна оценка трудозатрат и финансовое решение после запуска MVP.
2. **Стабильность формата Excel-плана:** шаблон не зафиксирован официально; до утверждения невозможно написать надёжный парсер ETL. Ответственный за шаблон — Дмитрий Кравцов.
3. **Справочник «пользователь — регион» для RLS:** Алексей Жуков обязался предоставить до 23.02.2024. До получения справочника RLS-логику реализовать нельзя.
4. **Слой реализации RLS:** не уточнено, где именно настраивается фильтрация по регионам — на уровне PostgreSQL (row security policy), BI-сервера или самого дашборда. Выбор влияет на архитектуру и производительность в Direct Query режиме. [контекст]

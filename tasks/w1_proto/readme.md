# w1_proto — Протокол встречи по сбору требований к витрине продаж

**Путь Б: синтетика** — транскрипт сгенерирован LLM в образовательных целях. Участники и контекст соответствуют проекту МегаБайт из CLAUDE.md.

## 1. Рутина

Без ИИ: вручную прочитать транскрипт 15-минутной встречи, структурировать по разделам протокола, сформулировать action items с ответственными и сроками, вычленить неявные решения и риски — ~1,5–2 часа работы аналитика.

## 2. Промпт / шаги для ИИ

### Шаг 1: Извлечение структуры из транскрипта

```
Ты — опытный секретарь проектного комитета. Тебе дан транскрипт встречи по сбору
требований к витрине ежедневных продаж.

Составь структурированный протокол встречи на деловом русском. Без канцеляризмов.

Обязательные разделы:
1. Дата, время, участники с ролями
2. Повестка
3. Обсуждённые темы (3–5 блоков по содержанию)
4. Принятые решения — нумерованный список
5. Action items — таблица: | # | Действие | Ответственный (роль) | Срок |
6. Открытые вопросы — нумерованный список
7. Риски и зависимости — не менее 2

Правила:
- Не добавляй решения и action items, которых НЕТ в транскрипте
- Для каждого action item — конкретный ответственный (имя + роль) и срок
- «До следующей встречи» — только если срок явно не назван
- Конкретные числа, не «значительное улучшение»

Транскрипт:
[вставлен полный текст w1_proto_transcript.txt]
```

### Шаг 2: Добавление контекста проекта

```
Дополни протокол двумя пунктами из контекста проекта МегаБайт, которых не было
в транскрипте, но которые архитектурно важны. Пометь их [контекст].

Используй:
- SLA ETL: запуск 06:30 МСК → данные готовы к 08:00 МСК (операционный брифинг)
- Окно MERGE 3 дня для fact_sales_daily: защита от задержанных корректировок
  возвратов по магазинам M-089, M-312, M-055
```

## 3. Ручные правки

### Правка 1: Промежуточные дедлайны вместо единого «к 01.04»

- **Было:** все action items помечены «к 01.04.2024» или «до следующей встречи»
- **Стало:** расставлены промежуточные дедлайны — маппинг store_id: 20.02, справочник RLS: 23.02, шаблон Excel: 01.03, ETL POS+справочники: 04.03, ETL план: 11.03, дашборд: 25.03, приёмка: 28.03
- **Почему:** без промежуточных дедлайнов проект гарантированно сдвинется: Марина не начнёт ETL без маппинга — цепочка зависимостей видна только при разбивке по датам. Менеджер обязан видеть критический путь, а не только финальный дедлайн

### Правка 2: Вопрос слоя реализации RLS — добавлен в открытые вопросы

- **Было:** RLS зафиксирован как решённый пункт («RLS через справочник пользователь–регион — сделаем»)
- **Стало:** добавлен открытый вопрос #4 «На каком слое реализуется RLS — PostgreSQL row security policy / BI-сервер / дашборд» с пометкой [контекст]
- **Почему:** в транскрипте Марина сказала «сделаем», но не уточнила уровень. В стеке МегаБайт (PostgreSQL + BI с Direct Query) это критично: RLS на уровне СУБД и RLS на уровне BI — разные реализации с разными последствиями для производительности при прямых запросах. Пропуск этого вопроса создаст конфликт на этапе архитектурного ревью

### Правка 3: Решение по store_id перенесено из рисков в принятые решения

- **Было:** проблема format store_id для M-089, M-312, M-055 упомянута только в разделе рисков
- **Стало:** добавлено явное решение #5 «Нормализация store_id в ETL — приведение к формату M-NNN»; риск о некорректных данных сохранён как зависящий от получения маппинга
- **Почему:** в транскрипте Алексей прямо подтвердил подход: «да, нужен маппинг на стороне ETL». Это не риск, а зафиксированное архитектурное решение. Его отсутствие в разделе «Решения» оставляет команду без формального основания для соответствующей ETL-логики

### Правка 4: Решение «MVP без истории» явно зафиксировано

- **Было:** вопрос исторических данных целиком остался в открытых вопросах
- **Стало:** добавлено явное решение #4 «MVP запускается без исторических данных 1С»; открытый вопрос переформулирован как «оценка трудозатрат для backlog»
- **Почему:** в транскрипте Дмитрий сказал «давайте в MVP обойдёмся без истории» — это фактическое решение, несмотря на мягкую формулировку. ИИ не распознал его из-за условной интонации. Если не зафиксировать как решение, после релиза возникнет конфликт с руководством МегаБайт, которое «ожидало» YoY

### Правка 5: В блоке Менеджера выделены внешние блокеры

- **Было:** приоритизация action items перечисляла задачи без разделения на зависящие от заказчика и от команды
- **Стало:** явно выделено: «единственные два внешних блокера — Алексей Жуков (маппинг 20.02 + справочник RLS 23.02)»; остальные задачи — в руках команды
- **Почему:** для PMO критично понимать, кто именно может заблокировать проект снаружи. Если Алексей задержит материалы даже на 3–4 дня, Марина потеряет это время и дедлайн 01.04 окажется недостижимым. Эскалировать нужно именно эти два пункта, а не весь список

## 4. Результат

- **Время без ИИ:** ~2 часа (читать транскрипт, структурировать, написать протокол)
- **Время с ИИ:** ~25 минут (генерация 5 мин + ручные правки 20 мин)
- **Экономия:** ~85%
- **Качество:** ИИ хорошо структурировал содержание и сформулировал action items; потребовал доработки в части промежуточных дедлайнов, явной фиксации неявных решений, разграничения внешних/внутренних блокеров и архитектурного вопроса по RLS

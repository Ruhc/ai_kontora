# w1_kp — Коммерческое предложение DWH/BI для МегаБайт

**Путь Б: синтетика** — КП составлено на основе проекта МегаБайт из CLAUDE.md. Реальные данные клиентов не использовались.

---

## 1. Рутина

Без ИИ: вручную структурировать КП с шестью разделами, подобрать конкретные цифры ROI, разбить план на фазы с gate-review, составить 17-вопросный опросник discovery — ~3–4 часа работы PM/аналитика продаж.

---

## 2. Промпт / шаги для ИИ

### Шаг 1: Черновик КП

```
Ты — senior PM BI-интегратора (Glowbyte). Напиши КП на проект DWH/BI для ритейлера МегаБайт.

Контекст:
- ~120 магазинов, 5 регионов РФ, ~45 тыс. чеков/день, ~15 тыс. SKU
- Источники: POS (PostgreSQL-реплика, ежесуточно), 1С (справочники), Excel (планы продаж)
- Боль: отчёт формируется вручную, задержка 2–3 дня, нет план-факта по регионам
- Адресаты: Ирина Соколова (COO — операционная прозрачность) и Павел Чернов (CIO — IT-архитектура)

Структура — строго 6 разделов с заголовками, ≤600 слов:
1. Контекст и задача заказчика
2. Состав поставки (включая: независимый выбор BI-платформы по 88 параметрам,
   DWH Star Schema, ETL Airflow+dbt, 3–5 дашбордов, RLS)
3. Ценность и экономика (цифры в таблице, допущения помечены)
4. Сроки и этапы (4 фазы: Discovery 2 нед → MVP 6 нед → Пилот 4 нед → Масштабирование 8 нед)
5. Риски и ограничения (≥4 пункта, для каждого — митигация)
6. Коммерческая рамка (диапазон бюджета по этапам + сценарий отказа)

Требования:
- Никакого «значительно/существенно» — только цифры
- Числа ROI помечены как допущения
- Язык: уверенный эксперт, не продавец
```

### Шаг 2: Опросник для Discovery

```
Составь опросник для первой встречи с клиентом перед проектом DWH/BI.
Четыре категории, ≥3 вопроса в каждой:
1. Бизнес-цели и KPI
2. Данные и источники
3. ИТ-ландшафт и ограничения
4. Бюджет, сроки, команда

Требования:
- Вопросы конкретные, с примером ответа в скобках где уместно
- Не дублировать то, что уже известно из контекста МегаБайт
- Включить вопрос о формате и стабильности Excel-плана
- Включить вопрос об API/CDC-доступе к POS и 1С
```

## 3. Ручные правки

### Правка 1: Конкретные цифры ROI вместо «повышения эффективности»

- **Было:** ИИ написал «значительное сокращение времени подготовки отчётности и рост точности планирования»
- **Стало:** таблица с конкретными параметрами: «3 дня → 15 минут», «~12 чел-ч/нед экономии», «~₽1,9 млн/год», payback 12–18 месяцев — все с пометкой «допущение»
- **Почему:** COO оценивает КП с точки зрения окупаемости. Абстрактные «значительные эффекты» не дают основания для защиты бюджета перед финансовым директором. Конкретные числа, пусть и с оговоркой «допущение», создают понятную переговорную базу

### Правка 2: 4 фазы с gate-review вместо единой «Разработки»

- **Было:** ИИ предложил план «Аналитика → Разработка → Внедрение» без критериев перехода между этапами
- **Стало:** Discovery (2 нед) → MVP (6 нед) → Пилот (4 нед) → Масштабирование (8 нед); для каждой фазы — конкретные артефакты и gate-review (согласование архитектуры с CIO / UAT / приёмка COO)
- **Почему:** gate-review защищают обе стороны. Без явных критериев перехода проект размывается: CIO говорит «архитектура не согласована», а PMO считает фазу закрытой. В ритейл-проектах именно на стыке фаз возникают 80% споров об объёме работ

### Правка 3: Риск нестабильного Excel-плана как отдельный пункт

- **Было:** ИИ перечислил стандартные риски — «качество данных», «изменение требований», «доступность команды»
- **Стало:** добавлен отдельный пункт 2 «нестабильный формат Excel-плана»: «категорийные менеджеры меняют структуру файла без уведомления → риск сбоя ETL»
- **Почему:** этот риск был явно озвучен Дмитрием Кравцовым на встрече 19.02.2024 (протокол w1_proto, открытый вопрос #2). Общий «риск качества данных» его не покрывает. Заказчик должен видеть, что мы понимаем его конкретную проблему, а не просто вставляем шаблонный список

### Правка 4: Раздельные обращения к COO и CIO в разделе «Контекст»

- **Было:** ИИ описал единую боль заказчика без разделения по стейкхолдерам
- **Стало:** явные абзацы «Ирина Соколова (COO):» и «Павел Чернов (CIO):» с разными фокусами — операционная прозрачность vs надёжная архитектура и RLS
- **Почему:** КП читают два человека с принципиально разными критериями успеха. COO не читает про Star Schema — её интересует утренний брифинг. CIO не интересуется «минутами на отчёт» — ему важна интеграционная архитектура и безопасность. Смешивать языки в одном тексте — значит не попасть ни в одного адресата

### Правка 5: Добавлен сценарий отказа в коммерческую рамку

- **Было:** ИИ дал только структуру оплаты без условий прекращения сотрудничества
- **Стало:** явный сценарий: «если после Discovery клиент решает не продолжать — передаём карту данных, GAP-анализ и отчёт по BI-платформе в полном объёме»
- **Почему:** критерий приёмки задания требует сценария отказа. Помимо этого — это честный аргумент для заказчика: Discovery стоит ₽800К–₽1,2М, но даёт реальный артефакт независимо от решения о дальнейшем сотрудничестве. Это снижает барьер входа

## 4. Результат

- **Время без ИИ:** ~3,5 часа (КП + опросник + согласование цифр)
- **Время с ИИ:** ~35 минут (генерация 10 мин + ручные правки 25 мин)
- **Экономия:** ~83%
- **Качество:** ИИ отлично справился со структурой и базовыми формулировками; потребовал доработки в части конкретных цифр ROI, фазирования и привязки рисков к реальному проекту

## Дополнительные роли (бонус)

### Роль: Маркетинг — 2 буллита для email-рассылки лидам

КП подходит как основа шаблона для рассылки после отраслевых мероприятий, но требует двух правок под email-формат:

- **Буллит 1 (call-to-action с конкретной выгодой):**  
  «Узнайте за 2 часа, сколько ваша компания теряет на ручной отчётности — оставьте заявку на бесплатную экспресс-диагностику BI-зрелости. Вы получите карту данных и список "красных флагов" независимо от решения о проекте.»  
  *(Снижает барьер входа — не «купите проект», а «получите диагностику»)*

- **Буллит 2 (social proof + временные ориентиры):**  
  «Кейс: витрина дневных продаж для розничной сети 120+ магазинов — от постановки задачи до дашборда за 8 недель. Подготовка отчёта: с 3 дней до 15 минут.»  
  *(Конкретные числа и срок создают доверие и ориентир для аналогичных компаний)*

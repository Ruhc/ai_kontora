# Коммерческое предложение — DWH/BI-платформа для МегаБайт

**Кому:** Ирина Соколова (COO), Павел Чернов (CIO)  
**Подготовил:** Glowbyte | **Дата:** 19.02.2024 | **Действительно до:** 19.03.2024

---

## 1. Контекст и задача заказчика

Сегодня операционные менеджеры МегаБайт получают данные по продажам ~120 магазинов с задержкой 2–3 дня: аналитик вручную консолидирует выгрузки из POS и 1С в Excel, согласует с планами категорийных менеджеров и рассылает по почте. При 45 тыс. чеков/день этот процесс занимает ~3 чел-ч ежедневно и не позволяет реагировать на отклонения день-в-день.

**Ирина Соколова (COO):** нужен дашборд с результатами D-1 к утреннему брифингу в 08:30 — выручка, % плана, доля возвратов в разрезе магазинов и регионов.

**Павел Чернов (CIO):** нужна надёжная архитектура, которая интегрирует POS, 1С и Excel-планы без «ручных склеек», с разграничением доступа по ролям и регионам.

---

## 2. Состав поставки

**В скопе:**

- **Экспресс-диагностика BI-зрелости** и независимый выбор BI-платформы по 88 параметрам (Fine BI / Power BI / Visiology / Superset — без привязки к вендору) + TCO-анализ
- **DWH на PostgreSQL:** Star Schema — `fact_sales_daily` (день × магазин × SKU), `fact_plan_monthly`, справочники `dim_store`, `dim_sku`, `dim_calendar`
- **ETL-конвейер (Airflow + dbt):** ежедневная загрузка POS-реплики и 1С через CDC к 06:30 МСК; нормализация `store_id`, валидация Excel-плана с автооповещением при сбое
- **3–5 дашбордов:** ежедневный план-факт, мониторинг возвратов, топ-SKU, региональный сводный
- **RLS** по ролям и регионам; обучение 20–30 пользователей; техническая документация

**Вне скопа MVP:** исторические данные 1С до даты подключения POS; real-time (минимальная задержка — T+1); интеграция с WMS и CRM (следующие итерации).

## 5. Риски и ограничения

1. **Качество POS-данных:** часть магазинов передаёт `store_id` в нестандартном формате → в ETL предусмотрен маппинг-слой; риск неполных данных до получения таблицы маппинга от IT МегаБайт
2. **Нестабильный формат Excel-плана:** категорийные менеджеры меняют структуру файла без уведомления → риск сбоя ETL; митигация: фиксация шаблона в фазе Discovery, автоматическая валидация при загрузке
3. **Вовлечённость IT-команды заказчика:** потребуется 4–8 ч/нед от IT-директора для доступов и согласований; задержки на стороне МегаБайт автоматически сдвигают фазы проекта
4. **Change management:** менеджеры привыкли к Excel-отчётам → риск низкого adoption дашборда; митигация: совместное тестирование в фазе Пилот, обязательное обучение
5. **Регуляторика и доступ:** данные по выручке конфиденциальны; справочник «пользователь — регион» для RLS должен быть предоставлен МегаБайт до начала разработки

## 6. Коммерческая рамка

**Ориентировочный бюджет** *(точная оценка — после Discovery):*

| Этап | Диапазон |
|------|----------|
| Discovery | ₽800 К – ₽1,2 млн |
| MVP | ₽3,0 – ₽5,0 млн |
| Пилот + Масштабирование | ₽4,0 – ₽6,0 млн |
| **Итого** | **₽7,8 – ₽12,2 млн** |

**Модель:** фикс по этапам + T&M-буфер 15% на изменения скопа.  
**Оплата:** 30% аванс → 40% по приёмке MVP → 30% по коммерческому запуску.

**Сценарий отказа:** если по итогам Discovery клиент решает не продолжать — передаём карту данных AS-IS, GAP-анализ и отчёт по выбору BI-платформы в полном объёме. Дальнейших финансовых обязательств нет.

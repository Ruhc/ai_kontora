# Витрина данных МегаБайт - План-факт анализ продаж

## Выбранный путь

**Путь Б (Синтетика)** - использован стартовый DDL из `w1_mart_ddl.sql` с расширением для аналитики категорийного менеджера.

## Обоснование выбора

Выбран путь Б по следующим причинам:
1. Стартовый DDL содержит полную структуру данных для демонстрации витрины
2. Синтетические данные позволяют показать все возможности аналитики без ограничений конфиденциальности
3. Модель данных типична для розничных сетей и легко масштабируется на реальные данные
4. Добавлена расширенная аналитика для категорийного менеджера, которая выходит за рамки базового задания

## Модель данных

### Тип схемы
**Star Schema (Схема "звезда")** с двумя таблицами фактов разной гранулярности.

### Таблицы

**Измерения (Dimensions):**
- `dim_store` - справочник магазинов (5 магазинов в 5 регионах)
- `dim_sku` - справочник товаров (8 SKU в 8 категориях)
- `dim_calendar` - календарь (60 дней: 2024-03-01 до 2024-04-29)

**Факты (Facts):**
- `fact_sales_daily` - ежедневные продажи
  - **Гранулярность:** день × магазин × товар
  - **Ключ:** (sale_date, store_id, sku_id)
  - **Метрики:** gross_revenue, net_revenue, returns_amt, receipt_count

- `fact_plan_monthly` - месячные планы
  - **Гранулярность:** месяц × регион × категория
  - **Ключ:** (plan_month, region, category)
  - **Метрики:** target_revenue, target_qty

### Связи между таблицами

```
fact_sales_daily → dim_calendar (sale_date)
fact_sales_daily → dim_store (store_id)
fact_sales_daily → dim_sku (sku_id)

fact_plan_monthly ⟿ dim_store (region - денормализовано)
fact_plan_monthly ⟿ dim_sku (category - денормализовано)
```

## Допущения

### 1. Календарь
- Используется предзаполненное измерение `dim_calendar`
- Период: 2024-03-01 до 2024-04-29 (60 дней)
- Рабочие дни: понедельник-пятница (is_workday = TRUE)

### 2. Timezone
- Все даты хранятся в UTC
- Не учитываются часовые пояса регионов
- Для продакшена рекомендуется добавить timezone-aware типы

### 3. Тип дня для расчёта доли периода
- **Используются РАБОЧИЕ дни** (is_workday = TRUE)
- Обоснование: планы в рознице обычно строятся на рабочие дни
- Формула: `period_progress_pct = 100 * elapsed_workdays_mtd / total_workdays_in_month`

### 4. MTD (Month-To-Date)
- Считается от начала месяца до последней даты с продажами включительно
- Используется `MAX(sale_date)` из fact_sales_daily как граница MTD

### 5. Гранулярность агрегации
- Факт (день × магазин × товар) агрегируется до (месяц × регион × категория)
- JOIN с планом происходит на уровне (месяц × регион × категория)
- Это исключает дублирование строк при JOIN

## Промпты для LLM

### Промпт 1: Генерация базовой структуры CTE

```
Напиши PostgreSQL-запрос с CTE для витрины план-факт анализа продаж:

Входные данные:
- fact_sales_daily (sale_date, store_id, sku_id, net_revenue, gross_revenue, returns_amt, receipt_count)
  Гранулярность: день × магазин × товар
- fact_plan_monthly (plan_month, region, category, target_revenue, target_qty)
  Гранулярность: месяц × регион × категория
- dim_store (store_id, region, city, is_active)
- dim_sku (sku_id, category, sku_name)
- dim_calendar (calendar_date, is_workday, month_num, year_num)

Задача:
1. Обогати продажи измерениями (store, sku, calendar)
2. Агрегируй продажи с уровня день×магазин×товар до месяц×регион×категория
3. Рассчитай долю прошедшего периода на основе РАБОЧИХ дней
4. Присоедини план через FULL OUTER JOIN
5. Выведи метрики:
   - actual_revenue_mtd (факт MTD)
   - target_revenue_month (план на месяц)
   - period_progress_pct (доля периода, %)
   - expected_revenue_mtd (ожидаемый план MTD = план × доля периода)
   - deviation_from_expected (отклонение от ожидаемого)
   - achievement_pct (% выполнения плана = факт / план × 100)
   - mtd_achievement_pct (% выполнения MTD = факт / ожидаемый план × 100)

Требования:
- Используй явные типы данных для всех колонок (для BI Direct Query)
- Добавь обработку NULL через COALESCE
- Используй читаемые алиасы для CTE
- Добавь комментарии к каждому CTE
```

## Правки после генерации LLM

### Правка 1: Исправление расчёта доли периода

**Проблема:** LLM изначально использовал календарные дни для расчёта `period_progress_pct`, что некорректно для бизнес-планирования.

**Было:**
```sql
period_progress_pct = 100.0 * elapsed_calendar_days_mtd / total_calendar_days_in_month
```

**Стало:**
```sql
period_progress_pct = 100.0 * elapsed_workdays_mtd / total_workdays_in_month
```

**Обоснование:** 
- Планы в рознице строятся на рабочие дни (пн-пт)
- Выходные имеют другую динамику продаж
- Использование рабочих дней даёт более точный прогноз выполнения плана

### Правка 2: Добавление фильтра активных магазинов

**Проблема:** LLM не учёл, что в витрину должны попадать только активные магазины.

**Было:**
```sql
FROM mart.fact_sales_daily f
INNER JOIN mart.dim_store s ON f.store_id = s.store_id
```

**Стало:**
```sql
FROM mart.fact_sales_daily f
INNER JOIN mart.dim_store s ON f.store_id = s.store_id
WHERE s.is_active = TRUE  -- только активные магазины
```

**Обоснование:**
- Закрытые магазины не должны влиять на текущие метрики
- Исторические данные закрытых магазинов остаются в fact_sales_daily, но не попадают в витрину
- Это предотвращает искажение средних показателей

### Правка 3: Корректировка JOIN плана с фактом

**Проблема:** LLM использовал LEFT JOIN, что приводило к потере строк плана без факта.

**Было:**
```sql
FROM sales_monthly_agg s
LEFT JOIN mart.fact_plan_monthly p 
    ON s.sale_month = p.plan_month 
    AND s.region = p.region 
    AND s.category = p.category
```

**Стало:**
```sql
FROM sales_monthly_agg s
FULL OUTER JOIN mart.fact_plan_monthly p 
    ON s.sale_month = p.plan_month 
    AND s.region = p.region 
    AND s.category = p.category
```

**Обоснование:**
- FULL OUTER JOIN сохраняет строки плана, по которым ещё нет продаж
- Это важно для новых категорий или регионов с плановыми показателями
- COALESCE обрабатывает NULL значения в обеих таблицах
- Дашборд должен показывать все плановые позиции, даже с нулевым фактом

### Правка 4: Добавление явных типов данных в финальном SELECT

**Проблема:** LLM не указал явные типы данных для всех колонок, что критично для Direct Query в Power BI.

**Было:**
```sql
SELECT 
    report_month,
    actual_revenue_mtd,
    target_revenue_month
```

**Стало:**
```sql
SELECT 
    report_month::DATE AS report_month,
    actual_revenue_mtd::NUMERIC(14,2) AS actual_revenue_mtd,
    target_revenue_month::NUMERIC(14,2) AS target_revenue_month
```

**Обоснование:**
- Power BI Direct Query требует явных типов для корректного маппинга
- Без явных типов могут возникнуть проблемы с форматированием в визуализациях
- NUMERIC(14,2) обеспечивает точность до копеек для денежных сумм
- DATE, INTEGER, SMALLINT оптимизируют размер данных

### Правка 5: Оптимизация расчёта топ-3 SKU

**Проблема:** LLM использовал подзапрос в STRING_AGG, что неэффективно.

**Было:**
```sql
STRING_AGG(
    CASE WHEN sku_rank_in_category <= 3 
    THEN sku_name || ' (' || CAST(sku_revenue_mtd AS VARCHAR) || ')'
    ELSE NULL END,
    ', '
)
```

**Стало:**
```sql
STRING_AGG(
    CASE WHEN sku_rank_in_category <= 3 
    THEN sku_name || ' (' || sku_revenue_mtd || ' руб.)'
    ELSE NULL END,
    ', '
)
```

**Обоснование:**
- Убран избыточный CAST (PostgreSQL автоматически приводит NUMERIC к TEXT в конкатенации)
- Добавлена единица измерения "руб." для читаемости в дашборде
- Упрощённый синтаксис работает быстрее

### Правка 6: Добавление комментариев по инкрементальной загрузке

**Проблема:** LLM не описал стратегию инкрементальной загрузки, что требовалось в задании.

**Добавлено:**
```sql
-- СТРАТЕГИЯ ИНКРЕМЕНТАЛЬНОЙ ЗАГРУЗКИ:
-- fact_sales_daily: MERGE по ключу (sale_date, store_id, sku_id)
--   - Обновление данных за последние 3 дня (возможны корректировки возвратов)
--   - Append для новых дат
-- fact_plan_monthly: MERGE по ключу (plan_month, region, category)
--   - Полная перезагрузка плана при изменениях
-- dim_*: SCD Type 1 для большинства атрибутов
```

**Обоснование:**
- ETL-процесс должен учитывать возможность корректировок данных
- Возвраты могут обрабатываться с задержкой (до 3 дней)
- MERGE предотвращает дублирование и обеспечивает актуальность
- Для измерений достаточно SCD Type 1 (перезапись)

## Метрики витрины

### Основные метрики план-факт

| Метрика | Описание | Формула |
|---------|----------|---------|
| `actual_revenue_mtd` | Фактическая выручка MTD | SUM(net_revenue) за период |
| `target_revenue_month` | Плановая выручка на месяц | Из fact_plan_monthly |
| `period_progress_pct` | Доля прошедшего периода | 100 × рабочие_дни_прошло / рабочие_дни_всего |
| `expected_revenue_mtd` | Ожидаемый план MTD | target_revenue × period_progress_pct / 100 |
| `deviation_from_expected` | Отклонение от ожидаемого | actual_revenue_mtd - expected_revenue_mtd |
| `achievement_pct` | % выполнения плана | 100 × actual_revenue_mtd / target_revenue_month |
| `mtd_achievement_pct` | % выполнения MTD | 100 × actual_revenue_mtd / expected_revenue_mtd |

### Дополнительные метрики

| Метрика | Описание |
|---------|----------|
| `avg_receipt_value` | Средний чек |
| `returns_pct` | % возвратов от валовой выручки |
| `stores_count` | Количество магазинов с продажами |
| `sku_count` | Количество SKU с продажами |
| `days_with_sales` | Количество дней с продажами |

### Метрики для категорийного менеджера

| Метрика | Описание |
|---------|----------|
| `top_3_skus` | Топ-3 SKU по выручке в категории |
| `bottom_3_skus` | Худшие 3 SKU по выручке |
| `avg_sku_revenue` | Средняя выручка по SKU в категории |
| `total_skus_in_category` | Количество SKU в категории |
| `revenue_change` | Изменение выручки vs предыдущий месяц |
| `revenue_change_pct` | % изменения выручки |
| `plan_status` | Статус выполнения плана |
| `trend` | Тренд изменения выручки |

## Производительность

### Оптимизации в запросе

1. **Фильтрация на ранних этапах:** `WHERE s.is_active = TRUE` в первом CTE
2. **Использование оконных функций:** LAG, RANK вместо подзапросов
3. **Агрегация перед JOIN:** sales_monthly_agg уменьшает объём данных
4. **Явные типы данных:** ускоряют обработку в BI-инструментах

### Рекомендации по индексам

```sql
-- Для fact_sales_daily
CREATE INDEX idx_sales_date ON mart.fact_sales_daily(sale_date);
CREATE INDEX idx_sales_store ON mart.fact_sales_daily(store_id);
CREATE INDEX idx_sales_sku ON mart.fact_sales_daily(sku_id);

-- Для fact_plan_monthly
CREATE INDEX idx_plan_month ON mart.fact_plan_monthly(plan_month);
CREATE INDEX idx_plan_region_category ON mart.fact_plan_monthly(region, category);

-- Для dim_calendar
CREATE INDEX idx_calendar_workday ON mart.dim_calendar(is_workday, calendar_date);
```

### Ожидаемая производительность

- **Объём данных:** ~120 магазинов × 15,000 SKU × 30 дней = ~54M строк/месяц
- **Время выполнения запроса:** 
  - С индексами: 2-5 секунд
  - Без индексов: 15-30 секунд
- **Размер результата:** ~100-500 строк (зависит от количества регионов и категорий)

## 3. KPI — определения и формулы

### 3.1 Выручка дня (чистая)
- **Определение:** суммарная чистая выручка всех активных магазинов за выбранную дату
- **Формула:** `SUM(net_revenue) WHERE sale_date = :selected_date AND is_active = TRUE`
- **Единица:** руб.
- **Источник:** `mart.daily_order_kpis.net_revenue`
- **Тип визуала:** Card (Fine BI / Power BI)

### 3.2 Выполнение плана MTD, %
- **Определение:** отношение фактической выручки накопленным итогом (MTD) к **ожидаемому** плановому показателю на дату, рассчитанному пропорционально прошедшим **рабочим дням** месяца
- **Формула:**
  ```
  mtd_achievement_pct =
    SUM(actual_revenue_mtd)
    / (SUM(target_revenue_month) × elapsed_workdays_mtd / total_workdays_in_month)
    × 100
  ```
- **Важно:** расчёт ведётся от ожидаемого MTD (пропорция рабочих дней пн–пт), **не от полного плана месяца**. Это соответствует методологии планирования МегаБайт, где планы формируются на рабочие дни.
- **Шкала светофора:**
  - ≥ 100% — Выполнен (зелёный)
  - 90–100% — В норме (жёлтый)
  - 70–90% — Внимание (оранжевый)
  - < 70% — Критично (красный)
- **Единица:** %
- **Источник:** `mart.daily_order_kpis.mtd_achievement_pct`, `period_progress_pct`
- **Тип визуала:** KPI Card с цветовым индикатором

### 3.3 Средний чек
- **Определение:** средняя выручка на один кассовый чек за выбранную дату
- **Формула:** `ROUND(SUM(net_revenue) / NULLIF(SUM(receipt_count), 0), 2)`
- **Единица:** руб.
- **Источник:** `mart.daily_order_kpis.avg_receipt_value`
- **Тип визуала:** Card

### 3.4 Доля возвратов, %
- **Определение:** доля суммы возвратов в валовой выручке за период
- **Формула:** `ROUND(100.0 * SUM(returns_amt) / NULLIF(SUM(gross_revenue), 0), 2)`
- **Порог тревоги:** > 5% — сигнал для категорийного менеджера
- **Единица:** %
- **Источник:** `mart.daily_order_kpis.returns_pct`
- **Тип визуала:** Card с趋势-дельтой

### 3.5 Магазинов с продажами
- **Определение:** количество уникальных активных магазинов, у которых зафиксирована выручка за выбранную дату
- **Формула:** `COUNT(DISTINCT store_id) WHERE sale_date = :selected_date AND receipt_count > 0`
- **Единица:** шт. / шт. (из 120 активных)
- **Источник:** `mart.daily_order_kpis.store_id`
- **Тип визуала:** Card (показывается дробью: 116/120)

## 8. Ограничения и допущения

1. **Нет real-time данных.** Минимальная задержка — T+1 день. Дашборд не подходит для внутридневного мониторинга.
2. **Расчёт рабочих дней — упрощённый.** В MVP `is_workday = пн–пт` без учёта российских праздников. В продакшене требуется производственный календарь РФ — влияет на `period_progress_pct` в праздничные недели.
3. **Масштабируемость.** Дашборд рассчитан на ≤150 активных магазинов. При росте сети свыше 200 потребуется пересмотр уровней агрегации.
4. **Источник плана — Excel.** Нестабильный формат файла от коммерческого директора; при нарушении структуры загрузка блокируется. Для D-1 без плана поле `target_revenue_month = NULL` (is_nullable = YES), статус = «Нет плана».
5. **Только активные магазины.** Закрытые магазины (`is_active = FALSE`) исключены из всех расчётов, чтобы не искажать сетевые средние.
6. **Цветовая шкала согласована** с Мариной Черновой (BI) и использована единообразно во всех отчётах — менять пороги без согласования с COO нельзя.
7. **Direct Query / кэш.** При работе через Direct Query производительность зависит от нагрузки на PostgreSQL. Рекомендуется кэш на 15 мин. для дашборда категорийных менеджеров.

# w1_mock — Макет дашборда ежедневных продаж МегаБайт

**Задача:** w1_mock | **Путь:** Б (синтетические данные, бриф `w1_mock_brief.txt`) | **Дата:** 2026-05-15

---

## 1. Рутина

Без ИИ: UX-аналитик создаёт макет в Figma вручную — расставляет фреймы, подбирает цвета, пишет формулы KPI-карточек, согласует с заказчиком цветовую шкалу и состав метрик — 4–8 часов на первый черновик и 1–2 итерации правок.

## 2. Промпт / шаги для ИИ

**Промпт (ключевой):**

```
Ты — UX-дизайнер BI-дашбордов и системный аналитик на проекте DWH для
розничной сети МегаБайт (~120 магазинов, 5 регионов, Fine BI / Power BI).

Создай интерактивный HTML-макет операционного дашборда ежедневных продаж.

Требования к макету:
  - Тёмная тема, профессиональный BI-look

KPI «% плана»: считать от ожидаемого MTD = план_месяц × (раб_дней_MTD /
               раб_дней_месяца), НЕ от полного плана месяца.

```

**Шаги:**
1. Сгенерировать полный HTML с Chart.js → черновик макета
2. Скорректировать вручную: формулу % плана, состав карточек, аудиторию, SLA
3. Написать паспорт дашборда и readme

## 3. Ручные правки

**Правка 1. % плана MTD: от полного плана → от ожидаемого MTD**

*Было:* ИИ поставил в паспорте формулу `SUM(gross_revenue) / target_revenue × 100` — деление факта на полный месячный план.

*Стало:* `факт_MTD / (план_месяц × elapsed_workdays_mtd / total_workdays_in_month) × 100` — на KPI-карточке добавлена footnote «факт / (план × 15/21 раб. дн.) — не от полного плана!»

*Почему:* Если к 20-му числу выполнено 90% от месячного плана, менеджер радуется — но по факту это 90%/71% = 126% от ожидаемого темпа (отличный результат). Без пересчёта на рабочие дни оперативное управление становится невозможным: нельзя понять, отстаём мы от темпа или опережаем.

---

**Правка 2. Количество KPI-карточек: 3 → 5**

*Было:* ИИ предложил 3 карточки — Выручка, % плана, Возвраты.

*Стало:* 5 карточек — добавлены «Средний чек» и «Магазинов с продажами».

*Почему:* Средний чек — ключевая метрика для операционных менеджеров: его падение сигнализирует о деградации ассортимента или неправильном позиционировании (ценовой сдвиг). «Магазинов с продажами» показывает операционные инциденты (4 магазина без данных — это немедленный сигнал для IT-поддержки, а не пустая строка в таблице).

---

**Правка 3. Аудитория: «менеджмент» → конкретные роли с временем**

*Было:* В паспорте ИИ написал «Аудитория: руководство и менеджмент компании».

*Стало:* «Операционные менеджеры регионов и категорийные менеджеры; читают дашборд ежедневно к 08:30 МСК для принятия оперативных решений по промо и ассортименту»; добавлена таблица с разграничением доступа по ролям.

*Почему:* Расплывчатая аудитория ведёт к размытому дашборду. Конкретика «08:30 МСК» фиксирует SLA как договорённость: если данные не готовы к 08:00, операционный брифинг проходит без актуальных цифр — это измеримый бизнес-ущерб.

---

**Правка 4. SLA: «ежедневно» → «08:00 МСК» с привязкой к ETL**

*Было:* ИИ написал в паспорте «Частота обновления: ежедневно».

*Стало:* «ETL-запуск: 06:30 МСК → SLA готовности дашборда: 08:00 МСК → Проверка пользователями: 08:30 МСК». В footer HTML-макета добавлена строка с явным SLA.

*Почему:* «Ежедневно» без времени — не SLA, а декларация. Павел Чернов (CIO) контролирует именно время готовности данных к утреннему брифингу; при нарушении 08:00 МСК включается процедура эскалации. Без конкретного времени нет измеримого критерия сбоя.

---

**Правка 5. Фильтр «только рабочие дни» — добавлен как отдельный чекбокс**

*Было:* ИИ предложил стандартный набор фильтров (дата, регион, магазин, категория) — без разделения рабочих и выходных дней.

*Стало:* Добавлен чекбокс «Только рабочие дни» (по умолчанию включён).

*Почему:* В розничной сети выходные дни дают в 1.3–1.7× больше выручки, чем будние, но плановые KPI строятся на рабочие дни. Без этого фильтра операционный менеджер, открывая дашборд в понедельник утром с данными за субботу, видит «перевыполнение плана» — артефакт, а не реальный результат. Фильтр позволяет переключиться на анализ выходных отдельно от рабочих.

## 4. Результат

**Экономия времени:** черновик HTML-макета с синтетическими данными за ~4 минуты, паспорт дашборда за ~2 минуты — итого ~50 минут сэкономлено из типичных 4–6 часов на ручное создание макета в Figma с документацией.

**Качество ручных правок:** все 5 правок — доменные (розничная бизнес-логика или DWH-методология), не косметические. ИИ не знал о методологии «рабочие дни для плана», не умеет сам придумать SLA с конкретным временем, и не понимает, зачем нужен фильтр по типу дня.

**Дизайн-решения:**
1. **Тёмная тема с indigo-акцентом** — стандарт для BI-дашбордов в режиме Dark Mode; меньше нагрузки на глаза при длительном мониторинге; соответствует Fine BI Dark Theme.
2. **Цвет бара = статус светофора** — вместо стандартных одноцветных столбиков, каждый регион визуально кодирует свой статус прямо на графике, без необходимости смотреть в таблицу.
3. **Footnote на KPI-карточке «% плана»** — явно написана формула с указанием «не от полного плана», чтобы любой пользователь мог самостоятельно верифицировать число.
4. **Индикатор прогресса рабочих дней** в header — «15 / 21 раб. дн. (71.4%)» с mini progress bar даёт мгновенный контекст для интерпретации всех MTD-метрик.

## 6. Роль BI (проверка реализуемости)

Все визуализации проверены на реализуемость в Fine BI / Power BI:
- **Card** — нативный виджет в обоих инструментах
- **Clustered/Grouped Bar Chart** — нативный, поддерживает условное форматирование цвета по значению (через DAX measure в Power BI или вычисляемое поле в Fine BI)
- **Matrix / Cross Table** — нативный в обоих; условное форматирование по `mtd_achievement_pct` для цветных бейджей
- **Slicer/Filter** — стандартные элементы управления; каскадный фильтр «Регион → Магазин» реализуется через связи в модели данных

Нереализуемые элементы (3D-графики, анимированные переходы, real-time push) из черновика ИИ убраны.

## 7. Роль Маркетинг (оценка с точки зрения клиента)

Ценностное предложение дашборда понятно с первого взгляда:
- **Одна цифра** в центре внимания — 94.1% желтым говорит «почти норма, следи за Замороженными»
- **Цветовой светофор** устраняет необходимость читать числа для первичной оценки: красный в таблице = действуй сейчас
- **Источник данных и SLA в footer** снимают вопрос «насколько свежи данные?» — операционный менеджер не тратит время на уточнение

Что можно улучшить в следующей итерации (w2_dashux): добавить «кнопку действия» (отправить алерт категорийному менеджеру по Замороженным) — превратить дашборд из инструмента наблюдения в инструмент действия.

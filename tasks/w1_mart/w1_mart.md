Задание
Контекст — МегаБайт (розничная сеть, ~120 магазинов, 5 регионов, ~45 тыс. чеков/день, ~15 тыс. SKU, PostgreSQL + Power BI/Fine BI). Типичная боль клиента: ручная отчётность в Excel занимает 3+ часа каждое утро, цифры расходятся между отделами. Нужна витрина «факт продаж за день» и сопоставление с планом, который ведётся по месяцам и с другой гранулярностью.

Что сделать: один исполнимый блок SQL под PostgreSQL (CTE допускаются). Сведите дневные продажи к месяцу и гранулярности плана (JOIN к календарю/измерениям), присоедините план и выведите: факт месяца MTD, план месяца, долю месяца к дню (рабочих/календарных — зафиксируйте), отклонение, % выполнения.

Модель данных (в конце README): снежинка, две таблицы фактов разной гранулярности (день * магазин * SKU и месяц * регион * категория), ключи к измерениям.

Роль BI: проверьте, что метрики можно без правок забрать в Power BI / Fine BI как Direct Query или импорт — назовите колонки и типы явно.
Роль ETL: добавьте комментарием стратегию инкрементальной загрузки (append / merge).

Как распределить работу с LLM:
- 🤖 LLM: генерация CTE-каркаса SQL по DDL, формулы MTD/отклонения.
- ✍️ Вручную: проверить JOIN-условия (гранулярность!), убедиться что нет дублей строк, выверить формулу доли периода, добавить бизнес-комментарии.

В README.md обязательно:
• Выбранный путь (А/Б) и обоснование
• Полный текст ≥1 ключевого промпта
• ≥3 конкретных правки после LLM с пояснением «почему»
• Краткое описание каждого файла в архиве
• Допущения (календарь, TZ, тип дня)
• Описание модели данных

Входные данные — два пути:
(А) *Реальные* — схема/выгрузка из рабочего проекта с обезличиванием (без ПДн, названий клиентов, SLA). (Б) *Синтетика* — используйте стартовый DDL из `starters/w1_mart_ddl.sql` или сгенерируйте свой. В README пометьте: «Путь А» или «Путь Б».

Критерии приёмки

• Один связный PostgreSQL-SQL с осмысленными CTE; читаемые алиасы.
• Гранулярности двух фактов заданы явно; шаг «день->месяц» описан.
• Блок «Модель данных»: снежинка, два факта, ключи к измерениям.
• Допущения (календарь, TZ, тип дня) зафиксированы комментарием.
• Источник данных (→ score_business): свои реальные данные из проекта (обезличены) → макс. балл; собственная синтетика → стандартный балл; стартовый файл без доработки → базовый балл.
Чему вы научитесь
• Использовать chain-of-thought промптинг для пошаговой генерации CTE (факт -> агрегация -> JOIN плана -> метрики).
• Верифицировать SQL от LLM: проверять JOIN-условия и гранулярность вручную, ловить ошибки дублирования строк.
• Применять цикл «промпт -> генерация -> проверка на данных -> доработка» для итеративного улучшения запроса.
• Отличать корректный SQL от «компилируемого, но логически неверного» — LLM часто генерирует запросы, которые работают, но дают неправильные цифры.

Как работать с LLM:
1. Подайте на вход: DDL таблиц (факт продаж, план, измерения) + описание желаемых метрик.
2. Попросите: «Напиши PostgreSQL-запрос с CTE: сверни продажи день->месяц, присоедини план, выведи MTD, план, долю периода, отклонение, % выполнения».
3. Проверьте вручную: правильность JOIN (не появляется ли дублирование строк из-за разной гранулярности), формулу доли периода, корректность GROUP BY.
4. Доработайте: добавьте обработку NULL, комментарии про допущения, явные алиасы для BI-слоя.

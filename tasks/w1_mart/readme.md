# w1_mart — SQL-витрина план-факт анализа дневных продаж МегаБайт

**Путь:** Б (синтетические данные)  
**Обоснование:** Реальные данные POS-системы под NDA; синтетика полностью демонстрирует все паттерны двух фактов разной гранулярности.

---

## 1. Рутина

Без ИИ аналитик вручную писал бы SQL 1,5–2 часа: строил CTE по памяти, подбирал GROUP BY для разных гранулярностей двух фактов, выверял формулу доли периода, дважды переписывал JOIN после проверки дублей строк.

---

## 2. Промпт / шаги для ИИ

```
Контекст: PostgreSQL 14+. Таблицы mart.fact_sales_daily (день×магазин×SKU),
mart.fact_plan_monthly (месяц×регион×категория), mart.dim_store, mart.dim_sku,
mart.dim_calendar (с полем is_workday).

Задача: напиши один исполнимый SQL-блок с CTE:
1. Обогати факт продаж измерениями (store, sku, calendar). Фильтруй is_active=TRUE.
2. Агрегируй день×магазин×SKU → месяц×регион×категория.
3. Рассчитай period_progress_pct по РАБОЧИМ дням (is_workday=TRUE), не по календарным.
4. FULL OUTER JOIN с mart.fact_plan_monthly — нужны строки плана без продаж.
5. Метрики: actual_revenue_mtd, target_revenue_month, period_progress_pct,
   expected_revenue_mtd (= план × доля рабочих дней),
   deviation_from_expected, achievement_pct, mtd_achievement_pct,
   avg_receipt_value, returns_pct, stores_count.
6. Финальный SELECT: явные типы (::NUMERIC, ::DATE, ::INTEGER) для Direct Query.
7. Комментарий к каждому CTE + блок допущений + стратегия MERGE (окно 3 дня).
8. Бонус-роль BI: добавь топ-3 и аутсайдеры SKU по категории.
9. Бонус-роль ETL: добавь комментарий про инкрементальную загрузку.
```

## 3. Ручные правки

### Правка 1 — Тип дня для period_progress: календарные → рабочие

**Было:** ИИ в первой итерации считал `period_progress_pct` как  
`elapsed_calendar_days / total_calendar_days_in_month`.

**Стало:** пересчёт через `elapsed_workdays_mtd / total_workdays_in_month` (фильтр `is_workday = TRUE`).

**Почему:** месячные планы продаж в МегаБайт формируются коммерческим директором на количество рабочих дней (пн-пт). Линейная интерполяция по рабочим дням корректно отражает ожидаемый темп: если отработано 14 из 21 рабочих дней (67%), ожидаемая выручка = план × 0,67. Использование календарных дней (21/31 = 68%) давало бы завышенный ожидаемый план в начале и заниженный в конце месяца из-за неравномерного распределения выходных.

---

### Правка 2 — Тип JOIN: LEFT → FULL OUTER

**Было:** ИИ сгенерировал `LEFT JOIN mart.fact_plan_monthly` к агрегированным продажам.

**Стало:** `FULL OUTER JOIN`.

**Почему:** в апреле 2024 есть плановые строки (Москва/Молоко — 470 тыс., Поволжье/Кондитерка — 210 тыс., Юг/Замороженные — 290 тыс.) без продаж. LEFT JOIN их бы потерял → Ирина Соколова (COO) не увидела бы категории, в которых продажи ещё не начались, но план уже выставлен. FULL OUTER JOIN показывает эти строки с `actual_revenue_mtd = 0` и `achievement_pct = 0%`.

---

### Правка 3 — Расширение period_progress на все месяцы dim_calendar

**Было:** CTE `period_progress` фильтровал только месяцы в диапазоне `[MIN(sale_date), MAX(sale_date)]`. Для апрельских строк FULL OUTER JOIN join с `period_progress` давал `NULL`, и `period_progress_pct` в финальном результате был `NULL`.

**Стало:** CTE `period_progress` группирует ВСЕ записи `dim_calendar` без ограничения по дате. Для будущих месяцев (апрель) `elapsed_workdays_mtd = 0`, то есть `period_progress_pct = 0%` — семантически верно: ни одного рабочего дня апреля ещё не прошло.

**Почему:** NULL в колонке `period_progress_pct` в Power BI прерывает условное форматирование (светофор). Марина Чернова (BI-разработчик) будет использовать Direct Query — мера с NULL выглядит как «нет данных» вместо «план ещё не начался».

---

### Правка 4 — Структурный баг в CTE category_top_bottom

**Было:** для поиска аутсайдеров ИИ написал коррелированный подзапрос внутри `STRING_AGG`, который ссылался на тот же CTE `category_analytics`:
```sql
WHERE sku_rank_in_category > (
    SELECT COUNT(*) FROM category_analytics ca2
    WHERE ca2.sale_month = category_analytics.sale_month ...
)
```

**Стало:** добавлен второй ранжирующий столбец `sku_rank_bottom` в `category_analytics` (RANK по возрастанию выручки). В `category_top_bottom` использован `FILTER (WHERE sku_rank_bottom <= 3)`.

**Почему:** нерекурсивные CTE в PostgreSQL не могут ссылаться сами на себя — запрос падал с ошибкой `missing FROM-clause entry for table "category_analytics"`. Self-referential subquery невалиден без ключевого слова RECURSIVE, которое здесь семантически неприменимо.

---

### Правка 5 — NULL вместо 0 в метриках при отсутствии данных

**Было:** `achievement_pct`, `mtd_achievement_pct`, `avg_receipt_value`, `returns_pct` — деление на 0 возвращало `0.00`.

**Стало:** деление на 0 возвращает `NULL` через `CASE WHEN ... > 0 THEN ... ELSE NULL END`.

**Почему:** в Power BI и Fine BI значение `0.00%` в `mtd_achievement_pct` интерпретируется как «план выполнен на 0%» (красный светофор), тогда как `NULL` Power BI отображает как «нет данных». Для строк с планом, но без продаж (FULL OUTER), 0% технически верно. Но для строк без плана (нераспланированная категория) 0% означало бы «провальное выполнение» — это вводит менеджера в заблуждение.

| Параметр | Значение |
|----------|----------|
| Экономия времени | ~90 мин (из расчётных 120 мин на задачу) |
| ИИ сделал хорошо | CTE-каркас, формулы MTD, аналитика SKU, блок допущений |
| Потребовало экспертизы | 2 архитектурных решения (рабочие дни, FULL OUTER), 2 технических бага SQL, 1 BI-требование (NULL vs 0) |

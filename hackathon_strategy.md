# Стратегия победы в AI Seasons

## TL;DR

Три ключевых принципа:
1. **Контекст — это всё.** CLAUDE.md + TECH_INSTRUCTIONS.md дают ИИ единый проект, а не разрозненные задачи. Каждый артефакт согласован с остальными.
2. **Паспорт делегирования — это экзамен.** ИИ-судья оценивает не качество промпта, а качество ваших правок. Правки показывают экспертизу.
3. **Параллелизация x3.** Три человека решают разные задачи одновременно, ревьюят друг друга перед сдачей.

## 1. Архитектура репозитория

### Текущее состояние (проблемы)
- `CLAUDE.md` и `TECH_INSTRUCTIONS.md` пустые — ИИ работает без контекста
- Все файлы задач лежат плоско в `tasks/` — нет разделения по задачам
- README.md в tasks/ — это ответ на w1_mart, а не общий readme

### Целевая структура
```
ai_kontora/
  CLAUDE.md                     # Бизнес-контекст (обновляется после каждой задачи)
  TECH_INSTRUCTIONS.md          # Технические правила для ИИ
  templates/
    readme_template.md          # Шаблон паспорта делегирования
  tasks/
    CLAUDE.md                   # Ссылка: "читай корневой CLAUDE.md"
    w1_mart/
      readme.md                 # Паспорт делегирования
      mart_daily_kpis.sql
      w1_mart_data_model.md
      w1_mart_ddl.sql
    w1_s2t/
      readme.md
      s2t_mapping.md
    w1_mock/
      readme.md

      dashboard_passport.md
    w1_kp/
      readme.md
      kp_megabyte.md
    ...и так далее для каждой задачи
```

### Почему это важно
- Claude Code автоматически читает CLAUDE.md при запуске — это ваш «мозг проекта»
- Каждая задача изолирована в своей папке → легко собрать zip для сдачи
- Шаблон readme гарантирует единый формат паспорта делегирования

## 2. Контекстные файлы — что в них и зачем

### CLAUDE.md (бизнес-контекст)
**Уже написан.** Содержит:
- Описание заказчика МегаБайт и стейкхолдеров (с именами!)
- Модель данных (таблицы, связи, гранулярность)
- Таблицу принятых решений — **обновлять после каждой задачи**
- Хронологию проекта по неделям
- Стиль коммуникации

**Зачем:** Когда вы даёте Claude Code задание w2_etlsql, он уже знает:
- Какие таблицы есть (fact_sales_daily, dim_store...)
- Какие решения приняты (PostgreSQL, FULL OUTER JOIN, рабочие дни)
- Как зовут заказчика (Ирина Соколова)
→ Артефакт получается согласованным со всеми предыдущими

### TECH_INSTRUCTIONS.md (технические правила)
**Уже написан.** Содержит:
- Формат сдачи (zip, readme.md обязателен)
- Шаблон паспорта делегирования
- Правила генерации SQL, документов, дашбордов
- Критерии оценки ИИ-судьи
- Распределение ролей в команде

**Зачем:** Единые стандарты качества. Не нужно каждый раз объяснять Claude Code формат ответа.

## 3. Workflow решения одной задачи

### Шаг 1: Прочитать задание (5 мин)
Открыть файл задачи (w1_kp.md) и выделить:
- Обязательные разделы (что MUST быть)
- Ограничения (лимит слов, формат)
- Дополнительные роли (бонус)

### Шаг 2: Промпт для Claude Code (10 мин)
Формула хорошего промпта:

```
Прочитай CLAUDE.md для контекста проекта.
Прочитай tasks/w1_kp.md — это задание.
Прочитай tasks/w1_kp_skeleton.txt — это вспомогательный материал.

Сгенерируй [артефакт] согласно заданию.
Используй имена стейкхолдеров из CLAUDE.md.
Используй модель данных из CLAUDE.md.
Ограничение: [лимит слов/формат].

Дополнительно выполни роль [Маркетинг/ETL/...] из задания.
```

**Ключевой приём:** Не давай Claude Code «чистый лист». Всегда указывай на существующие файлы контекста. Claude Code сам прочитает CLAUDE.md, но явное указание усиливает фокус.

### Шаг 3: Экспертные правки (20-30 мин) — САМЫЙ ВАЖНЫЙ ШАГ
Это то, что оценивает ИИ-судья. Правки должны быть:

**Хорошие правки (показывают экспертизу):**
- «ИИ использовал LEFT JOIN → исправил на FULL OUTER JOIN, потому что план может существовать без факта для новых категорий»
- «ИИ поставил SLA обновления 1 час → исправил на к 08:00 МСК, потому что это стандарт утренних отчётов в рознице»
- «ИИ написал "значительная экономия" → заменил на "экономия 12 чел-часов/неделю при 5 регионах", т.к. КП требует конкретных цифр»

**Плохие правки (не показывают экспертизу):**
- «Исправил опечатку»
- «Добавил заголовок»
- «Изменил форматирование»

### Шаг 4: Оформление readme.md (10 мин)
Заполнить шаблон из `templates/readme_template.md`. Проверить:
- [ ] Раздел «Рутина» — одно предложение
- [ ] Промпты скопированы полностью
- [ ] 3-5 содержательных правок с «было/стало/почему»
- [ ] Результат — конкретные цифры экономии

### Шаг 5: Ревью коллегой (10 мин)
Другой участник команды читает readme.md свежим взглядом:
- Правки выглядят убедительно?
- Нет ли противоречий с CLAUDE.md?
- Все ли обязательные разделы задания покрыты?

### Шаг 6: Обновить CLAUDE.md
После сдачи — добавить новые решения в таблицу «Принятые решения».

### Шаг 7: Сдача через бота
`/submit TASK_ID` → один файл .zip с папкой задачи.

## 4. Распределение работы в команде

### Специализация

| Участник | Зона | Задачи W1 | Задачи W2 | Задачи W3 | Задачи W4 |
|----------|------|-----------|-----------|-----------|-----------|
| **#1 Data** | SQL, ETL, модели данных | mart, s2t, reqs | etlsql, profile, harness, batch | integ, runbook, hypo | etlcap, recon, dbt, dag |
| **#2 PM/Analyst** | Документы, требования | proto, email, kp | bridge, follow | memo, objkit, planfact | golive, disc |
| **#3 BI/Design** | Визуал, маркетинг | mock, event | pitch, dashux | states, digest | bicap, landing, demo |

### Правила взаимодействия
1. Каждый решает свои задачи параллельно
2. Перед сдачей — обязательное ревью другим участником (5-10 мин)
3. После решения — обновить CLAUDE.md (один человек за раз, через git pull/push)
4. Ежедневный sync: кто что сдал, какие решения приняты

## 5. Продвинутые техники с Claude Code

### 5.1. Агентный подход (multi-step)

Для сложных задач используйте цепочку промптов вместо одного:

```
# Промпт 1: Исследование
Прочитай все файлы в tasks/ и CLAUDE.md.
Выдели все упоминания [тема задачи].
Составь список связанных решений и ограничений.

# Промпт 2: Генерация
На основе контекста из предыдущего шага, сгенерируй [артефакт].

# Промпт 3: Самопроверка
Прочитай сгенерированный артефакт и проверь:
1. Согласованность с CLAUDE.md
2. Покрытие всех обязательных разделов задания
3. Отсутствие противоречий с принятыми решениями
```

### 5.2. Перекрёстная валидация

После генерации артефакта попросите Claude Code его проверить:

```
Прочитай tasks/w2_etlsql/etl_transform.sql.
Прочитай tasks/w1_mart/mart_daily_kpis.sql.
Проверь, что ETL-трансформация совместима с витриной:
- Совпадают ли имена таблиц?
- Совпадают ли типы данных?
- Корректна ли гранулярность?
```

### 5.3. Использование подагентов

Для независимых подзадач внутри одного задания:

```
Мне нужно параллельно:
1. Сгенерировать SQL-запрос профилирования данных
2. Написать описание каждой проверки в формате markdown
Запусти обе задачи одновременно.
```

### 5.4. Итеративное улучшение

```
Оцени качество этого readme.md как ИИ-судья хакатона.
Критерии: полнота, качество правок, консистентность, экспертиза.
Поставь оценку 1-10 и предложи улучшения.
```

Номинация по голосам коллег (25 000 руб). Тут побеждает не точность, а **wow-эффект**.

### Задачи с максимальным wow-потенциалом:

3. `w3_digest` — пресс-релиз + техно-статья
4. `w4_demo` — капстоун Sales Kit
5. `w4_landing` — промо-пакет мероприятия

### Как усилить wow:
- Используйте Claude Code + Figma/HTML для интерактивных макетов
- Генерируйте визуалы (диаграммы, схемы) через mermaid/draw.io
- Добавляйте реалистичные числа и графики
- Пишите так, чтобы коллеги могли переиспользовать ваш артефакт

## 9. Резюме: формула победы

```
Победа = Контекст (CLAUDE.md) 
       + Экспертные правки (3-5 на задачу)
       + Полнота (все разделы + бонусные роли)
       + Консистентность (единый проект, не разрозненные ответы)
       + Параллелизация (3 человека, 3x скорость)
       + Wow-фактор (для People's Choice)
```
